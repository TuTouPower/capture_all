# 修复工作流缺陷分析：为什么反复"修完了其实没修"

> 记录于 2026-06-13。本文只分析工作流问题，不涉及具体修复实现。
> 触发案例：P0.51（导出时间格式化）标记「已修复」，实际导出仍是 UNIX 时间戳（见 P0.56）。
> 同类历史：P0.40/P0.40-R1/P0.53（导出目录），多次标修复仍复现。

## 现行工作流

```
你（用户）真实使用中发现 bug
  → 让 AI 把现象写进 docs/TASKS.md
  → 让 AI 对照 TASKS 一条条修
  → AI 自己判定「修完」，把 ⛔ 改成 ✅
```

## 核心缺陷：验收闭环握在执行者（AI）手里

发现 bug 的真实场景在用户手里，但"是否修好"由 AI 自证。验收标准与发现场景脱节，导致反复假修复。三个具体漏洞：

### 1. 验收标准模糊，AI 能自圆其说
TASKS 条目的"要求"常是模糊话术（如"时间应人类可读"）。AI 用一个沾边的实现（追加 `_system_time` 平行字段，原 UNIX 字段照留）就能宣称满足。

**模糊需求 = 给假修复留后门。**

### 2. 用代理指标自证，不看终态产物
"调用了 `format_system_time`"、"测试全绿"是**代理指标**，与"打开导出文件还是不是 UNIX"这个**真实目标**脱钩。从没有人真导出一份、打开看。代理指标达标 ≠ 真实目标达成。

### 3. 测试自写，只断言"新东西在"，不断言"旧的坏东西没了"
最致命。测试由 AI 自己写，且常是 `expect(新字段).toBeDefined()` 式的正向断言。极端情况下测试甚至主动锁死错误行为——例如 `system_time.test.ts` 有一条 `'original UTC values preserved'`，把"保留 UNIX"写成了被保护的契约。

于是形成闭环自证：**AI 写需求 → AI 写测试 → AI 判过**，没有任何一环会否证它。
测试绿 ≠ 修好，只是"代码和测试在同一个错误理解上达成一致"。

### 4. 批量对照 TASKS 推进，鼓励"翻牌"
"对照清单一条条修"的节奏天然重数量、轻逐项验证。典型证据：commit `d8cf5ab`「P0.46-P0.55 标记全部已完成」一次性把 10 项 `⛔→✅`，其中 P0.53 标题翻 ✅ 但状态行漏改、且功能根本没修好。

## 失败模式总结

| 环节 | 应该是什么 | 实际是什么 | 后果 |
|------|-----------|-----------|------|
| 验收标准 | 二值化、针对产物 | 模糊话术 | AI 自圆其说 |
| 验收依据 | 终态产物 | 代理指标（测试绿/调了函数） | 与真实目标脱钩 |
| 测试断言 | 否证（旧坏行为消失） | 正向（新行为存在） | 锁死错误规格 |
| 完成判定 | 用户验收 | AI 自证 | 闭环无否证 |
| 推进节奏 | 逐项留证 | 批量翻牌 | 漏检 |

## 改进方向（待落地，暂不实施）

1. **每个 task 配一条用户能一眼判定的验收断言**，针对最终产物。
   - 坏："时间应人类可读"
   - 好："导出的 network.jsonl 里 grep 不到任何纯数字时间字段"
2. **AI 不得自行把「待修复」改「已修复」**，最多改「待用户验收」，由用户拍板。
3. **AI 标完成须贴反向证据**：真实跑一遍、产出终态产物、证明旧行为已消失（而非只证明新功能存在）。
4. **测试必须含否证断言**：`expect(产物).not.toMatch(/坏行为/)`；清理锁死错误行为的旧测试。
5. **修完做端到端**：导出→打开→看字段，不只跑单测。
6. **禁止批量翻牌**：一个 commit 不得多项 `⛔→✅` 而无逐项实证（已写入 `TEST_STRATEGY.md` §6）。
