# P0.41 Response Body 采集时序竞态分析

> 日期：2026-06-12
> 状态：已确认根因，未修复

---

## 1. 背景

### 1.1 Capture All 是什么

Capture All 是一个 Chrome 浏览器扩展（MV3），用于采集用户浏览网页时产生的全部数据：网络请求、控制台日志、鼠标/键盘事件、页面导航、Cookie 变化等。采集结果导出为 JSON 文件，供 AI Agent 或人工分析。

核心功能之一是 **Response Body Capture**：不仅记录每个 HTTP 请求的 URL、状态码、请求头等元数据，还要拿到响应体内容（HTML、JSON、CSS、JS 等文本内容）。这是分析「用户看到了什么页面内容」的关键数据。

### 1.2 Chrome 扩展如何抓取网络请求

Chrome 扩展有两套独立的 API 来监听网络请求，它们各自有优缺点：

**API 1：`chrome.webRequest`**

- 扩展的标准网络监听 API
- 能拿到：URL、方法（GET/POST）、状态码、请求头、响应头、请求体
- **拿不到响应体**（这是 Chrome 的安全限制）
- 能监听所有 tab 的请求，不限于单个 tab
- 事件在请求完成时触发，时序可靠

**API 2：Chrome DevTools Protocol (CDP)**

- Chrome 的调试协议，和 Chrome DevTools（F12）用的是同一套
- 通过 `chrome.debugger.attach` 连接到目标 tab
- 能拿到：完整的请求/响应信息，**包括响应体**（`Network.getResponseBody`）
- **只能 attach 到一个 tab**，无法同时监听多个 tab
- 事件通过 WebSocket 推送，有网络延迟

### 1.3 双路径采集架构

由于 webRequest 拿不到响应体，CDP 只能监听一个 tab，Capture All 同时使用两套 API：

```
┌─────────────────┐     ┌─────────────────┐
│  chrome.webRequest│     │  CDP (Network.*) │
│  所有 tab 的请求  │     │  当前活跃 tab     │
│  无响应体         │     │  有响应体         │
└────────┬────────┘     └────────┬────────┘
         │                       │
         ▼                       ▼
┌────────────────────────────────────────┐
│         network_capture.ts             │
│                                        │
│  ① webRequest 请求完成时               │
│     尝试匹配一个 CDP 事件              │
│     匹配成功 → 合并响应体 → 导出       │
│     匹配失败 → 延迟等待 1500ms         │
│                                        │
│  ② CDP Network.loadingFinished 时      │
│     调用 Network.getResponseBody        │
│     尝试关联到延迟等待的 webRequest     │
└────────────────────────────────────────┘
```

### 1.4 匹配机制

webRequest 和 CDP 是两个独立的事件流，需要把它们「配对」才能把响应体挂到正确的请求上。匹配依据是：URL（去掉查询参数）+ HTTP 方法 + 状态码 + 时间戳（±2000ms 窗口）。

系统维护一个 `cdp_request_meta` Map，存储所有已收到的 CDP 请求元数据。当 webRequest 请求完成时，在这个 Map 中查找匹配的 CDP 条目。

### 1.5 延迟机制（Deferred）

CDP 事件可能比 webRequest 事件晚到。为了处理这种时序差异，系统设计了延迟机制：

1. webRequest 请求完成，找不到匹配的 CDP 事件 → 不立即标记 `not_enabled`
2. 创建一个 deferred 条目，记录该请求正在等待哪些 CDP 候选
3. 等待最多 1500ms（`DEFERRED_TIMEOUT_MS`）
4. 如果在此期间 CDP 响应体到达 → 关联成功 → 导出响应体
5. 如果超时 → 标记 `not_enabled`（无响应体）

---

## 2. 问题现象

导出 JSON 中，通过 webRequest 路径采集的请求大量缺少响应体：

| 采集 | webRequest 请求数 | `not_enabled` | 占比 |
|------|------------------|---------------|------|
| 第一次（15 秒） | 145 | 144 | 99.3% |
| 第二次（15 秒） | 114 | 112 | 98.2% |

同一采集中，CDP 路径的请求（`capture_method: "extension_cdp"`）响应体采集正常：69 条 `captured`，0 条 `not_enabled`。

这说明 CDP 本身是工作的，问题出在 webRequest → CDP 的关联环节。

---

## 3. 根因分析

### 3.1 主因：匹配函数删除了后续查找需要的数据

`handle_completed` 函数在 webRequest 请求完成时执行，流程如下：

```
步骤 1: find_matching_cdp_request(url, method, status, timestamp)
        在 cdp_request_meta 中查找匹配的 CDP 条目
        找到匹配 → 删除该条目 ← 问题根源
        但此时 CDP 响应体（body）尚未到达 → 无法立即返回
        继续往下

步骤 2: find_cdp_candidates(url, method, status)
        再次在 cdp_request_meta 中查找候选
        步骤 1 已删除匹配条目 → 候选数为 0

步骤 3: 创建 deferred 条目
        pending_cdp_ids 为空（因为步骤 2 找到 0 个候选）
        设置 1500ms 超时

步骤 4: 1500ms 后超时
        没有候选 → try_resolve_deferred 永远不会被触发
        → response_body_status = "not_enabled"
```

**关键代码**（`src/background/network_capture.ts`）：

```typescript
// 步骤 1: 尝试匹配
const matched_cdp_id = find_matching_cdp_request(url, method, status, timestamp);

if (matched_cdp_id) {
    cdp_request_meta.delete(matched_cdp_id);  // ← 删除了匹配条目
    const body_result = cdp_body_results.get(matched_cdp_id);
    if (body_result) {
        // 响应体已就绪 → 直接返回（正常路径）
        return;
    }
    // 响应体未就绪 → 不 return，继续往下
    // 但 cdp_request_meta 条目已被删除
}

// 步骤 2: 再次查找候选（此时步骤 1 的条目已不存在）
const candidates = find_cdp_candidates(url, method, status);  // → 0 个候选

// 步骤 3: deferred 条目无候选 → 必然超时
const pending_cdp_ids = new Set(candidates);  // → 空集
setTimeout(() => {
    // 步骤 4: 超时 → not_enabled
}, 1500);
```

### 3.2 次因：页面加载时 CDP 事件天然晚于 webRequest

即使没有上述删除问题，页面首次加载时也存在时序竞态：

```
时间线：
t=0ms   浏览器开始加载页面
t=1ms   webRequest 捕获到请求（chrome.webRequest 事件触发）
t=5ms   CDP attach 完成，开始监听
t=10ms  webRequest 请求完成 → handle_completed 触发
        此刻 cdp_request_meta 为空（CDP 事件还在传输中）
t=15ms  CDP Network.requestWillBeSent 事件到达
        但 deferred 条目已创建且 pending_cdp_ids 为空
t=1510ms deferred 超时 → not_enabled
```

日志证据：

```
21:56:17 body_deferred {url:"fonts.gstatic.com/...", cdp_meta_count:0, candidate_count:0}
         ↑ webRequest 请求完成，此刻 cdp_request_meta 为空
21:56:17 CDP body capture enabled {tab_id:1793063526}
         ↑ CDP 刚刚 attach，事件还没到
```

### 3.3 两个匹配函数互相干扰

系统中有两个匹配函数，共用同一个 `cdp_request_meta` 数据源：

| 函数 | 用途 | 匹配维度 | 副作用 |
|------|------|----------|--------|
| `find_matching_cdp_request` | 精确匹配（含时间窗口） | URL + method + status + 时间 ±2000ms | **删除匹配条目** |
| `find_cdp_candidates` | 宽泛匹配（无时间窗口） | URL + method + status | 无副作用 |

当 `find_matching_cdp_request` 先运行并删除条目后，`find_cdp_candidates` 看到的数据已不完整。

## 4. 日志证据

### 4.1 统计数据

第二次采集的完整日志（355 行）中：

| 日志类型 | 数量 | 含义 |
|----------|------|------|
| `body_not_enabled_deferred_timeout` | 112 | 延迟超时 → not_enabled（最终失败） |
| `body_deferred` | 113 | 进入延迟等待（其中 1 条后来被成功 resolve） |
| `cdp_match_miss` | 31 | find_matching_cdp_request 无匹配 |
| `deferred_resolving` | 1 | 延迟条目被成功解析（仅 1 条） |
| `body_captured` | 1 | 完整匹配 + 有响应体 |
| `body_not_enabled_immediate` | 0 | 没有走「CDP 未启用」路径 |

关键发现：112 条 `not_enabled` 全部走延迟超时路径，0 条走「CDP 未启用」路径。说明 CDP 配置正确、已正确 attach，问题完全在匹配环节。

### 4.2 时序证据

```
21:56:17 body_deferred {cdp_meta_count:0, candidate_count:0}
         → CDP 事件未到达，0 候选

21:56:17 CDP body capture enabled {tab_id:1793063526}
         → CDP 刚 attach

21:56:19 body_deferred {cdp_meta_count:27, candidate_count:0}
         → CDP 事件已到达（27 条元数据），但候选仍为 0
         → 原因：find_matching_cdp_request 已删除匹配条目

21:56:19 cdp_match_found_but_no_result {cdp_id:"79180.30"}
         → 匹配到了 CDP 条目，但响应体尚未到达

21:56:24 body_not_enabled_deferred_timeout {candidate_count:0, cdp_meta_at_timeout:40}
         → 超时，此刻有 40 条 CDP 元数据，但候选仍为 0
```

### 4.3 匹配失败原因分布

```
cdp_match_miss {cdp_meta_count:27, method_miss:27, url_miss:0}
cdp_match_miss {cdp_meta_count:33, method_miss:30, url_miss:3}
```

`method_miss` 占绝大多数：CDP 的 `Network.requestWillBeSent` 事件包含该 tab 上所有类型的网络请求（含 CORS 预检、ping、WebSocket 握手等），而 webRequest 记录的 `type` 字段（`xmlhttprequest`、`stylesheet` 等）与 CDP 的 `type`（`Fetch`、`Stylesheet` 等）格式不同，导致 method 维度匹配率低。

## 5. 修复方案

### 方案 A：保留已匹配 CDP ID 作为延迟候选（推荐）

**问题**：`find_matching_cdp_request` 找到匹配后立即删除 `cdp_request_meta` 条目，导致后续 `find_cdp_candidates` 看不到该条目。

**修复**：当匹配成功但响应体未就绪时，不删除 `cdp_request_meta` 条目，而是将该 CDP ID 直接传入延迟条目的 `pending_cdp_ids`。

```typescript
if (matched_cdp_id) {
    const body_result = cdp_body_results.get(matched_cdp_id);
    if (body_result) {
        // 响应体已就绪 → 正常返回
        cdp_request_meta.delete(matched_cdp_id);
        send_to_background(...);
        return;
    }
    // 响应体未就绪 → 保留条目，记住匹配 ID
    matched_but_pending_id = matched_cdp_id;
}

// 创建 deferred 条目时，将匹配 ID 加入候选集
const candidates = find_cdp_candidates(...);
const pending_cdp_ids = new Set(candidates);
if (matched_but_pending_id) {
    pending_cdp_ids.add(matched_but_pending_id);  // ← 保证有候选
}
```

**效果**：当 CDP 响应体通过 `Network.loadingFinished` 到达时，`try_resolve_deferred` 能通过 `pending_cdp_ids` 找到对应的延迟条目并关联。

**风险**：低。不改变匹配逻辑，只修复延迟候选集的完整性。

### 方案 B：延迟删除（替代方案）

将 `cdp_request_meta.delete()` 的调用从 `find_matching_cdp_request` 匹配成功时，延迟到响应体确认可用时。这样 `find_cdp_candidates` 仍能看到完整数据。

**风险**：低。需要确保延迟条目超时时也能正确清理 `cdp_request_meta`。

### 方案 C：CDP-first 架构（长期重构）

完全不用 webRequest 做响应体关联，改为 CDP 事件驱动：

1. CDP `Network.requestWillBeSent` → 创建请求记录
2. CDP `Network.responseReceived` → 更新状态码/响应头
3. CDP `Network.loadingFinished` → 获取响应体
4. webRequest 仅用于补充 CDP 未覆盖的信息（请求体、tab_id）

**优点**：彻底消除时序竞态，响应体采集可靠。
**缺点**：架构大改。CDP 只能 attach 到一个 tab，多 tab 场景需要为每个 tab 独立管理 CDP session（当前 P0.39 已解决跨 tab 切换，但每个 tab 独立 session 是更大的改动）。

---

## 6. 建议

1. **短期**（推荐立即实施）：方案 A 或方案 B。改动量小（~20 行），预期将 `not_enabled` 从 ~98% 降至接近 0%。
2. **中期**：统一两个匹配函数的查找逻辑，避免共用状态互相干扰。
3. **长期**：评估 CDP-first 架构的可行性，彻底消除双路径时序依赖。
