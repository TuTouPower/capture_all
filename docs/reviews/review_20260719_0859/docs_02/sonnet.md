无。

## 中低优先级问题

### M1. 模板缺少 task 元数据字段

- **位置**: `docs/templates/task/spec.md` 全文
- **现象**: 模板无 task ID、owner、branch、创建日期、状态等元数据占位符。对比 `docs/tasks/T008_label_routing/spec.md` 实际使用示例，该文件顶部包含 task ID 和链接，但模板未提供统一格式。
- **影响**: 每次创建 task spec 时，元数据格式依赖创建者自行决定，可能导致不一致。`docs/tasks/index.md` 记录 ID 和状态，但 spec 本身无法反向溯源。
- **建议**: 在 `# Task spec` 标题下方添加元数据块占位符，如 `Task: T{NNN} | Owner: {name} | Branch: {branch} | Status: {status}`。
- **置信度**: 中
- **优先级**: 中

### M2. 验收标准缺少格式指导

- **位置**: `docs/templates/task/spec.md:17`
- **现象**: 验收标准仅提供一个 checkbox 示例 `- [ ] {可独立验证的完成条件。}`，未说明应写多少条、粒度如何把握。
- **影响**: 创建者可能写过少（1 条笼统标准）或过多（拆分过细）。CLAUDE.md 开发原则要求"specs driven"和"可独立验证的完成条件"，但模板未将这些要求具象化。
- **建议**: 添加简短指导注释，如 `<!-- 每条标准应可独立验证，建议 2-5 条 -->`，或在占位符中给出多条示例结构。
- **置信度**: 中
- **优先级**: 中

### M3. 范围与非范围缺少示例

- **位置**: `docs/templates/task/spec.md:9,13`
- **现象**: 范围和非范围各只有一个占位符行。对于不熟悉项目的创建者，难以判断应写到什么粒度。
- **影响**: 可能导致范围描述过于宽泛或遗漏关键排除项。
- **建议**: 在占位符中添加 1-2 个示例，或在模板末尾添加注释说明写作要点。例如范围示例 `{修改 X 模块的 Y 行为}`，非范围示例 `{不涉及 Z 模块的重构}`。
- **置信度**: 低
- **优先级**: 低

## 改进建议

### S1. 添加变更日志占位符

当前模板无版本追踪机制。若 spec 在开发过程中被修订，无法记录变更历史。建议添加可选的 `## 变更记录` 节，或在元数据中包含 `Last updated: {date}` 字段。task `log.md` 已承担进展记录职责，但 spec 本身的修订与 task 进展是不同关注点。

### S2. 依赖与约束的"无"处理

模板指示"无则写'无'"，这是合理的。但可考虑在模板中直接预填 `- 无` 并注释 `<!-- 如有依赖请替换 -->`，减少创建时的认知负担。当前写法已足够清晰，此为锦上添花。

### S3. 模板与 CLAUDE.md 的交叉引用

CLAUDE.md 开发原则提到"specs driven：spec 和 plan 先行，一起写完交用户一次性审核"。模板本身未体现这一流程约束。建议在模板顶部添加一行注释：`<!-- 创建后与 plan.md 一同提交审核 -->`。当前模板简洁性优先，不加也可接受。

## 不确定项

### U1. 模板占位符风格选择

模板使用 `{中文描述}` 作为占位符，而非 `{english_placeholder}` 或 `TODO:` 标记。这是有意的中文化设计决策还是偶然？与代码中常见的英文占位符风格不同，但对中文用户更友好。无需修改，仅确认一致性。

- **置信度**: 低
- **行动**: 无需修改，确认为设计决策即可。

### U2. 模板是否应包含"参考"节

部分项目模板包含 `## 参考` 或 `## 相关链接` 节，用于引用设计文档、issue、PR 等。当前模板未提供此节。若 task 经常需要引用外部资料，可考虑添加。但保持模板最小化也是合理选择。

- **置信度**: 低
- **行动**: 按项目实际使用频率决定，当前无需修改。
