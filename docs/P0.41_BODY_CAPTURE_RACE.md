# P0.41 Response Body 采集时序竞态分析

> 日期：2026-06-12
> 状态：已确认根因，未修复
> 关联：P0.39 CDP retry 跨 tab 假成功

---

## 现象

导出 JSON 中 `web_request` 路径的请求大量 `response_body_status: "not_enabled"`，CDP 路径正常。

| 采集 | web_request 总数 | not_enabled | 占比 |
|------|-----------------|-------------|------|
| 第一次 | 145 | 144 | 99.3% |
| 第二次 | 114 | 112 | 98.2% |

`extension_cdp` 路径 0 条 `not_enabled`，body capture 本身工作正常。

---

## 根因

### 核心问题：`find_cdp_candidates` 删除已匹配的 CDP 元数据，导致 deferred 条目永远无候选

`handle_completed`（`network_capture.ts`）的处理流程：

```
1. find_matching_cdp_request(url, method, status, timestamp)
   → 找到匹配 → 从 cdp_request_meta 删除该条目 ← 问题根源
   → 无 body result → 落入步骤 2

2. find_cdp_candidates(url, method, status)
   → 步骤 1 已删除条目 → candidate_count: 0

3. 进入 deferred 等待（1500ms）
   → pending_cdp_ids 为空 → try_resolve_deferred 永远不会触发

4. 1500ms 超时 → not_enabled
```

**关键代码路径**（`network_capture.ts:596-640`）：

```typescript
// 步骤 1: find_matching_cdp_request 找到匹配
if (matched_cdp_id) {
    cdp_request_meta.delete(matched_cdp_id);  // ← 删除！
    // body result 不可用 → 不 return，继续往下
}

// 步骤 2: find_cdp_candidates 再次查找
const candidates = find_cdp_candidates(...);  // ← 步骤 1 已删除，0 候选

// 步骤 3: deferred 无候选 → 必然超时
const timer = setTimeout(() => {
    send_to_background(build_network_event(pending, details, null, 'not_enabled'));
}, DEFERRED_TIMEOUT_MS);
```

### 次要问题：页面加载时 CDP 事件晚于 webRequest

日志证据：

```
21:56:17 body_deferred {url:"fonts.gstatic.com/...", cdp_meta_count:0, candidate_count:0}
21:56:17 CDP body capture enabled {tab_id:1793063526}
```

页面加载期间，webRequest `handle_completed` 在 CDP `Network.requestWillBeSent` 事件到达之前触发。此时 `cdp_request_meta` 为空，`find_cdp_candidates` 必然返回 0。

即使 CDP 事件稍后到达，deferred 条目的 `pending_cdp_ids` 已为空集，`try_resolve_deferred` 无法关联。

### 次要问题：`find_matching_cdp_request` 与 `find_cdp_candidates` 匹配标准不一致

`find_matching_cdp_request` 使用时间窗口（±2000ms）+ URL + method + status 四维匹配。
`find_cdp_candidates` 使用 URL + method + status 三维匹配（无时间窗口）。

但两者共用 `cdp_request_meta` 作为数据源。步骤 1 删除条目后，步骤 2 看不到已删除的条目。

```

21:56:17 CDP body capture enabled {tab_id:1793063526}          ← CDP 刚 attach

```

### cdp_match_miss 分析

```
cdp_match_miss {cdp_meta_count:27, method_miss:27, url_miss:0}
cdp_match_miss {cdp_meta_count:33, method_miss:30, url_miss:3}
```

`method_miss` 占绝大多数：CDP 的 `Network.requestWillBeSent` 事件包含所有请求类型（含预检请求、ping 等），与 webRequest 的 `type` 字段不一定对应。当 `cdp_request_meta` 中存在大量 method 不匹配的条目时，`find_cdp_candidates` 命中率进一步降低。

---

## 修复方案

### 方案 A：保留已匹配 CDP ID 作为 deferred 候选（推荐）

**核心思路**：当 `find_matching_cdp_request` 找到匹配但无 body 时，将该 CDP ID 传入 deferred 的 `pending_cdp_ids`。

```typescript
if (matched_cdp_id) {
    const body_result = cdp_body_results.get(matched_cdp_id);
    if (body_result) {
        // 有 body → 立即返回
        cdp_body_results.delete(matched_cdp_id);
        cdp_request_meta.delete(matched_cdp_id);
        send_to_background(...);
        return;
    }
    // 无 body → 不删除 cdp_request_meta，传入 deferred
    matched_but_pending_id = matched_cdp_id;
}

// deferred 阶段
const candidates = find_cdp_candidates(...);
const pending_cdp_ids = new Set(candidates);
if (matched_but_pending_id) {
    pending_cdp_ids.add(matched_but_pending_id);  // ← 保证有候选
}
```

**效果**：当 CDP body 到达时（`Network.loadingFinished` → `try_resolve_deferred`），deferred 条目有候选可匹配。

**风险**：低。不改变匹配逻辑，只修复 deferred 候选集的完整性。

### 方案 B：deferred 阶段按 URL 二次关联（备选）

当 `pending_cdp_ids` 为空时，不立即超时，而是在 `Network.loadingFinished` 事件到达时按 URL 反向查找 deferred 条目：

```typescript
// Network.loadingFinished 处理中
for (const [key, entry] of deferred_web_requests) {
    if (entry.pending.url_base === cdp_meta.url_base) {
        // 尝试关联
    }
}
```

**风险**：中。URL 重复的请求可能错误关联。

### 方案 C：CDP-first 架构（长期）

完全不用 webRequest 做 body 关联，改为 CDP 事件驱动：

1. CDP `Network.requestWillBeSent` → 创建 pending 请求
2. CDP `Network.responseReceived` → 更新 status/headers
3. CDP `Network.loadingFinished` → 获取 body
4. webRequest 仅用于补充 CDP 未覆盖的信息（request body、tab_id）

**优点**：消除时序竞态，body capture 可靠。
**缺点**：架构大改，CDP 只能 attach 到一个 tab，多 tab 场景需要为每个 tab 独立 CDP session。

---

## 建议

1. **短期**：实施方案 A，修复 `find_cdp_candidates` 候选缺失问题。预期将 `not_enabled` 从 ~98% 降至接近 0%。
2. **中期**：将 `find_matching_cdp_request` 的删除操作延迟到 body 确认可用后，避免两个匹配函数互相干扰。
3. **长期**：评估 CDP-first 架构的可行性，彻底消除 webRequest ↔ CDP 时序依赖。
