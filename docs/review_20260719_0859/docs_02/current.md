## 中低优先级

### 1. 验收标准占位未要求“验证方式与可观察结果”

- 位置：`docs/templates/task/spec.md:15-17`，`## 验收标准`。
- 现象：模板只要求“可独立验证的完成条件”，未要求每条标准写明验证层级、操作或命令、预期可观察结果，也未提醒行为型标准必须验证终态产物。项目工作流要求 task spec 成为开发、agent-verify 和双路 review 的共同判定依据；`docs/guides/test.md` 还要求终态产物优先、通道对齐和否证断言。实际 spec 已出现只写“全绿”、结构存在或 grep 结果，但不总能表达用户行为和证据边界的情况。
- 影响：不同 implementer、verifier、reviewer 可能用不同代理指标解释同一验收项；功能可在单测或源码断言通过时被误判完成，行为型需求缺少黑盒闭环。验收勾选后也难追溯具体通过条件。
- 建议：把占位改为至少包含“条件／验证方式／预期结果”，并提示行为型 AC 使用用户可观察终态，清理或替换类 AC 至少包含一条旧行为消失的否证条件。例如：`- [ ] {完成条件} → 验证：{命令、测试层级或人工步骤} → 预期：{可观察结果}`。
- 置信度：高。
- 级别：MEDIUM。

### 2. 模板未显式要求写出用户确认决策与待确认问题

- 位置：`docs/templates/task/spec.md:3-21`，整体章节结构。
- 现象：模板只有背景、范围、非范围、验收标准、依赖与约束，没有“已确认决策”或“待确认问题”。项目采用 specs driven，spec 与 plan 需要先交用户审核；实际 `T008_label_routing/spec.md` 已自行增加“决策（用户确认）”，说明非显然需求确有记录决策来源的需要。
- 影响：有多种合理解释时，作者容易直接把假设写进范围或 AC，后续 reviewer 无法区分用户确认结论与实现者推断；跨轮次接手时也可能重复讨论，或按错误假设实施。
- 建议：增加可选章节 `## 已确认决策` 与 `## 待确认问题`，并要求无事项写“无”。待确认问题未闭合前，不应把对应实现项视为已批准范围。
- 置信度：中高。
- 级别：MEDIUM。

### 3. “依赖与约束”提示过宽，未形成项目硬约束核对清单

- 位置：`docs/templates/task/spec.md:19-21`，`## 依赖与约束`。
- 现象：占位只笼统提示“前置依赖、平台、安全或兼容性约束”。Capture All 存在多项高影响不变量：Bridge 绑定与 token 边界、IndexedDB 升级不丢数据、单活跃采集、HTML 转义、password 不采集、脱敏与截断分离、停止前 flush、生成物与密钥边界等。模板未要求作者判断哪些不变量受影响，也未提示数据迁移、旧数据兼容、权限与用户数据风险。
- 影响：task spec 可能只列直接依赖，漏掉跨模块不变量；review 虽可再读取 blueprint，但 spec 本身无法明确声明本 task 必须保持或验证哪些约束，增加安全、隐私和数据兼容回归遗漏概率。
- 建议：把该节拆成或补充提示：`前置依赖`、`受影响的业务不变量/安全边界`、`数据与向后兼容`、`平台/权限限制`；要求逐项写具体约束或“无”，并引用对应 blueprint 段落。
- 置信度：高。
- 级别：MEDIUM。

### 4. 范围占位未要求同步声明代码、测试与文档交付物

- 位置：`docs/templates/task/spec.md:7-13`，`## 范围` 与 `## 非范围`。
- 现象：模板只提示“包含什么／明确不做什么”，未提示按代码、测试、文档、迁移或产物划分。项目开发循环要求测试先行，并在每轮检查和更新受影响文档；实际 task spec 写法不一致，部分 spec 明列代码与测试，部分只列文件搬迁，文档更新或测试覆盖散落到 plan、AC 或完全缺失。
- 影响：spec 审批时难判断交付是否完整；实现者可能把测试或文档同步视为隐含工作，reviewer也缺少明确范围基线，容易发生实现完成但测试、guide、blueprint 候选未纳入 task 的情况。
- 建议：在范围占位中提示至少检查 `代码/配置`、`测试与黑盒验证`、`文档`、`数据迁移/兼容`、`构建或发布产物`；不适用项写“无”，避免强制创建空工作。
- 置信度：中高。
- 级别：MEDIUM。

### 5. 标题没有 task ID 与 slug 占位，降低文档自描述性

- 位置：`docs/templates/task/spec.md:1`，`# Task spec`。
- 现象：标题未包含 `{TNNN}`、标题或 slug。实际 spec 已自行使用多种标题格式，如 `Task spec — T008 label_routing`、`Task spec — T007 test_tree_reorg (Phase 4)`；模板没有统一约束。
- 影响：文件被单独引用、导出、搜索或脱离目录上下文显示时，无法仅凭标题识别所属 task；标题格式漂移也降低批量审阅和自动索引稳定性。
- 建议：改为 `# Task spec — {TNNN} {slug}`，可另加一行简短结果标题；保持 snake_case slug 与 `docs/tasks/TNNN_slug/` 一致。
- 置信度：高。
- 级别：LOW。

### 6. 未提示一个 task 只能承载一个独立可验证结果

- 位置：`docs/templates/task/spec.md:7-17`，范围与验收标准之间。
- 现象：`AGENTS.md` 明确要求一个 task 对应一个独立、可验证结果，互不依赖的结果应拆 task；模板没有把这一关键粒度规则带入填写现场。
- 影响：作者可能在一个 spec 中聚合多个可独立交付目标，导致范围膨胀、AC 相互无依赖却共用状态、分支和 finalization，增加部分完成、批量翻牌和回退困难。
- 建议：在“范围”前增加一句填写说明：若包含多个可独立验收结果，应先拆分 task；验收标准可有多条，但共同服务同一结果。
- 置信度：中高。
- 级别：LOW。

## 建议

1. 优先增强“验收标准”占位：固定要求完成条件、验证方式、预期可观察结果；加入行为终态与否证条件提示。
2. 增加“已确认决策／待确认问题”，让用户批准结论和实现者假设可区分。
3. 细化“依赖与约束”提示，强制检查受影响业务不变量、安全边界、数据兼容与平台限制。
4. 扩展“范围／非范围”填写指引，明确检查代码、测试、文档、迁移和产物，但允许不适用项写“无”。
5. 统一标题为 `# Task spec — {TNNN} {slug}`，并在模板提示单 task 单结果。
6. 保留当前五个核心章节及简洁体量；不建议把 plan 的步骤、风险与回退重复搬入 spec，避免双真相。
